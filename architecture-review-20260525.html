Problem: auth.config.ts registers a namespaced config block nobody reads — consumers bypass the namespace and hit Zod-validated flat keys directly. The registerAs wrapper is a shallow module whose interface is never used.

Solution: Delete auth.config.ts (and the unused app.config.ts, database.config.ts wrappers). The Zod envSchema is already the single source of truth. Keep JWT config reads as flat configService.get('JWT_SECRET').

  • locality: env shape lives in one file — validate-env.ts
  • deletion test: deleting auth.config.ts concentrates config truth, doesn't spread it
  • interface shrinks: one validation schema, not two parallel sources

Separate credential projection from user retrieval

Dependency category: in-process

Strong
src/users/users.service.ts • src/auth/auth.service.ts • src/users/dto/create-user.dto.ts

Before

flowchart LR
    A["AuthService\nvalidateUser()"] --> B["UsersService\nfindUserByUsername()"]
    B -->|"returns User {id,name,username,password}"| A
    style B fill:#fef2f2,stroke:#dc2626

Leak: password hash crosses the module seam

After

flowchart LR
    A["AuthService"] -->|"findUserByUsername()"| B["UsersService"]
    B -->|"UserProfile {id,name,username}"| C["Callers"]
    B -->|"UserCredential {id,password}"| A
    style B fill:#0f172a,stroke:#0f172a,color:#e2e8f0

Problem: findUserByUsername() returns the password hash to every caller. The users module interface leaks authentication internals — any new consumer of user retrieval must understand they're holding a secret. No seam between profile data and credential data.

Solution: Split into two retrieval methods: findUserProfileByUsername() returns {id, name, username}, and a new private-ish findUserCredentialByUsername() returns {id, password} for the auth module only. Or push credential comparison entirely into the users module behind a verifyCredentials(username, password): boolean interface.

  • leverage: callers get profile data without touching secrets
  • locality: credential comparison logic concentrates in one module
  • interface deepens: callers learn less, get the same behaviour

Extract Google Docs parser from AppService

Dependency category: in-process

Worth exploring
src/app.controller.ts • src/app.service.ts • src/common/

Before

After

Problem: generatePatternFromURL() is a 50-line HTML-table parser living in the root AppService. It has nothing to do with the application's purpose. It's guarded by JWT as an apparent test endpoint, but its implementation has no seam — can't test parsing logic without hitting the network.

Solution: Extract to src/common/parsers/pattern-parser.ts with a pure parseFromHTML(html: string) interface. The controller handles the URL fetch, the parser handles the extraction. parseFromHTML is testable with inline HTML strings; no mocks needed.

  • leverage: one parser, pure interface — testable with static HTML
  • locality: pattern logic lives in one module, not mixed with app concerns
  • deletion test: deleting the parser doesn't touch auth, users, or health

    style PI fill:#fef2f2,stroke:#dc2626

After

flowchart TD
    CD["CreateUserDto\n{name, username, password}"] --> US["UsersService.createUser()"]
    US --> PR["PrismaClient"]
    style CD fill:#0f172a,stroke:#0f172a,color:#e2e8f0

Standalone DTO; service maps to Prisma shape

Problem: CreateUserDto implements Prisma.UserCreateInput ties the HTTP contract to Prisma's generated types. A Prisma schema change (rename a field, add a relation) ripples into the API surface. The DTO carries concerns from two seams: HTTP validation and ORM shape.

Solution: Drop the implements Prisma.UserCreateInput. Define CreateUserDto as a standalone class. The UsersService.createUser() maps the DTO fields explicitly to the Prisma shape. This is a few lines of mapping for a stronger seam.

  • leverage: API surface doesn't change when the ORM schema changes
  • locality: Prisma shape knowledge stays inside the service
  • one adapter at each seam: DTO speaks HTTP, Prisma speaks database

Test AuthService in isolation with fake UsersService

Problem: Zero test files exist despite Jest infrastructure being scaffolded. The highest-value target — AuthService — depends on UsersService (which hits PrismaService) and JwtService. Every test would need a full NestJS module setup with real or deep-mocked dependencies. No in-process seam exists for substituting a fake UsersService.

Solution: Inject UsersService via its interface so tests supply a fake. Same for JwtService. Write AuthService unit tests against a fake that returns predetermined users. The fakes are one adapter; the real implementations are another — a real seam emerges.

  • leverage: one test suite exercises the full auth decision tree
  • locality: auth logic errors caught at the auth module boundary
  • two adapters justify the seam: real Prisma-backed + in-memory fake

Top recommendation

Add test seams to the auth pipeline

Tackle this first because it has the highest leverage: tests are the gating factor for every other refactor. Without a test seam, deepening any module is guesswork. AuthService is the narrowest interface with the richest behaviour — the perfect place to start. Once tested, the rest of the candidates become safe to collapse, extract, and delete.
